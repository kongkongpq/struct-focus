## 1. 现有设计的不合理之处

### 1.1 "确定性优先"是幻想——code agent 不需要确定性

Struct Agent 的核心宣言之一（ARCHITECTURE.md L82）：**确定性优先**。

这是整个项目最根本的**方向性错误**。

理由：
- **LLM 本身是不确定的**。temperature > 0、随机采样、模型权重更新——每一步都引入非确定性。你在确定性沙箱里跑一个非确定性引擎，得到的系统整体仍然是概率性的。
- **代码开发的路径不是确定性的，但结果是可验证的**。同一个需求有 N 种实现路径，路径选择本身就是创造性的、概率性的。追求"每次跑同一个 prompt 得到同一个结果"在 coding agent 场景下是**反目标**——你真正需要的是"每次都能得到正确结果"，而非"每次得到同一个结果"。**但**：编译结果、测试通过/失败、类型检查——这些是确定性的。确定性在**验证层**（而非执行层）是正确且有价值的。
- **确定性 ≠ 可审计**。可审计性可以通过完整的 Action/Observation 事件流 + 最终结果的正确性验证来保证（见 §2.2），不需要强制执行路径确定。

> 对标：OpenHands V1 明确宣布"默认无状态"（single source of truth = ConversationState），不做确定性保证，但通过 event-sourcing 保证**完全可重放**。SWE-agent 的多轨迹重试机制明确拥抱"同输入多输出，选最优解"。Claude Code 的 subagent 并行天然是非确定性的。

**建议**：把"确定性优先"替换为"**可追溯性优先**"——保证每一步 Action/Observation 完整记录、可重放、可审计，但不追求执行路径确定。这既解放了 LLM 的创造力（允许多次尝试），又守住了底线（知道发生了什么）。

### 1.2 上下文架构的"静态分层"不适应动态推理

哈佛 I/D 分离是 Struct Agent 最引以为傲的设计——但它是**静态的、线性的、单向的**：

```
I-Context (系统提示 + 记忆，稳定可缓存)
     ↓ 注入
D-Context (逐轮对话，Git 版本化)
```

问题：

**a) I-Context 和 D-Context 的边界是人为划定的，不是"推理需要"划定的。**
模型在推理过程中会自然地在"全局知识"（I）和"局部上下文"（D）之间切换。I/D 分离的本意是"稳定的前缀可以缓存"，但这只对 API 调用的前缀缓存有意义——**对 agent 自身的信息处理没有任何帮助**。Agent 关心的不是"这段内容能不能缓存"，而是"这段内容对我当前决策有没有用"。

**b) D-Context 的 Git 版本化是过度工程。**
把每次对话状态存为 Git commit，这听起来很聪明，但实际上：
- Commit 粒度是什么？一次工具调用？一个推理步骤？一次 LLM 请求？
- 如果你真的需要 "revert" 上下文（比如模型走进死胡同），你应该直接丢弃那部分 D-Context 并重新注入更紧凑的摘要——而不是做一个 `git revert`。
- Git 的 diff 语义对自然语言对话毫无意义——"这一行改了"是代码的概念，不是推理链的概念。

**c) 六原语（focus/forget/reflect/remember/recall）是正确的方向，但实现是"被动工具"而非"主动策略"。**
当前六原语是 6 个 tool，等着模型去调用。但模型什么时候会想起来调用 `reflect`？通常是"出问题了才想起来"。**正确的做法**是 agent 框架主动在关键节点（token 逼近阈值、连续错误、阶段切换）**自动触发**上下文管理——而不是指望模型自觉。

> 对标：Claude Code 的紧凑化是**框架层自动触发**的，不是模型调用的。Aider 的 RepoMap 是在每次 LLM 调用前**自动注入**的，不需要模型请求。SWE-agent 的 ACI 工具输出长度是**框架强制有界**的，不靠模型自觉。

### 1.3 五阶段提示词是"流水线思维"——代码开发不是流水线

```
explore → plan → execute → verify → summarize
```

这个线性五阶段的假设本身就有问题：

**a) 代码开发是高度非线性的。**
- 执行时发现新信息 → 需要回到探索阶段
- 验证时发现错误 → 可能是规划就有问题
- 很多时候 explore 和 execute 是同时发生的（边读边改）

**b) 阶段判断靠"近 3 步启发式"是脆弱的。**
`PhaseDetector` 用近 3 步的工具使用模式推断阶段——这相当于盲人摸象。模型可能在同一轮里先 explore 一个文件、再 execute 一个编辑、最后 verify 跑测试。三合一的时候你判它是哪个阶段？

**c) 每个阶段的"行为引导"实际上是束缚。**
"当前阶段：探索，优先用 file_read/file_list/code_search"——如果模型在探索阶段就发现了明显的 bug，它应该直接修，而不是先"规划"再"执行"。硬性的阶段划分降低了响应速度。

> 对标：SWE-agent 的 ReAct 循环是"think → act → observe"的通用模式，没有阶段概念，每次迭代根据观察自主决策。OpenHands 的 CodeAct 是"收到观察 → 产出动作"，不预设阶段。Anthropic 的 ReVeal（ICLR 2026）用 "generation ↔ verification" 交替循环，是更自然的节奏。

**建议**：放弃线性五阶段，改为**无阶段 ReAct + 上下文自适应注入**：框架在关键条件触发时（token 逼近、连续失败、新文件发现）自动向 context 注入简短的"情境提示"（不是阶段指令），让模型自主决策。

### 1.4 "工具定义 20 个全量暴露"是工具设计的反面教材

SWE-agent 的 ACI 设计原则第一条就是"**每次动作的信息量适中，不返回几千行，也不返回一行**"。Struct Agent 反其道而行：

- 20 个工具全量暴露给模型 → 每步 function-calling 要在 20 个选项里选 → 噪声大、易误用
- `file_read` 返回全文 → 一个小文件可能就 5000 tokens，大文件直接溢出
- 没有"当前文件/当前行"的状态跟踪 → 每次操作都得重新指定路径

**更根本的问题**：Struct Agent 的工具设计是**"把人类用的工具原样给 AI"**——file_read 就是 `cat`、file_list 就是 `ls`、shell_exec 就是 `bash`。SWE-agent 的核心洞察是"**人类工具和 AI 工具应该是不同的**"——AI 需要：
- 有界输出（view 限 100 行，有 scroll 操作）
- 显式状态（当前文件、当前行号、历史操作）
- 自动验证（编辑后 lint，不过就回滚）

### 1.5 Granularity Controller 的契约系统有根本性设计缺陷

GranularityController（500+ 行代码）是 Struct Agent 最复杂的子系统，包含了复杂度检测、LLM 拆解、契约注册、接口桩生成、冲突检测、拓扑排序、同层并行、漂移重跑、组合验证——整套流程。

**已确认**：`granularity-integration.ts` L66-137 实现了 `createGranularityController()` 工厂，能创建真实的子 StructFocus；`structfocus-agent.ts` L886-887 在 `granularity: true` 时确实调用 `controller.execute()`。**所以这不是死代码**——它是一个完整实现但**默认关闭**（`granularity` 默认 `false`）的 opt-in 特性。

但问题不在"是否接线"，而在**契约系统本身的根本性设计缺陷**：

**a) 契约声明是静态的，而代码修改是动态的。**
在拆解时就声明 `exports/imports/modifies`，但实际执行中 LLM 可能会修改未声明的文件。静态契约假设"我们能提前知道每个子任务会改哪些文件"——这在真实代码开发中几乎不成立。

**b) 漂移检测是事后的，而非预防性的。**
`verifyActuals` 在执行后检查，发现未声明的修改时只能"标记受影响的任务重跑"——但此时文件已被修改，下游子 Agent 可能已经基于过时的文件状态执行完毕。正确的做法是在执行前做**乐观锁**（基于文件哈希的冲突检测），而非事后补救。

**c) 接口桩生成是 best-effort 的空壳。**
代码注释写了"仅在声明了导出时生成桩"，但实际生成的桩只是占位符（如 `function foo() { throw new Error("Not implemented") }`），下游子 Agent 无法真正编译/运行——这意味着子 Agent 之间的"并行"实际上是假的，因为下游任务在等待上游真正实现完成前无法工作。

**d) 默认关闭本身就是问题。**
一个 500 行的子系统默认不启用，意味着：
- 大部分代码路径未被测试覆盖（`granularity: false` 时不走 Controller 逻辑）
- 这是一个"理论上很完善，但实践上不敢默认开启"的设计——这本身就是设计缺陷的信号

> **自审查更正**：原版本报告错误地声称 `createAgent` "从未实现"、"execute 从未调用"。经代码验证，两者均已实现并接线——错误在于默认关闭和设计缺陷，而非死代码。感谢严格的自审查发现此问题。

> 对标：OpenHands V1 的 SDK 直接不做多 Agent 编排——它认为单个 CodeAct agent 已经够强，多 Agent 的协调开销 > 收益。Anthropic 的 Subagent 做的是"结果压缩"而非"契约声明"——子 Agent 跑完后只返回摘要，不暴露内部细节。

### 1.6 工具数量不是问题，问题是每个工具都没有为 LLM 优化

Struct Agent 暴露了 20 个工具。批评"工具太多"是一个容易产生的误解——实际上**工具数量和工具质量是两个独立维度**。

**SWE-agent 的 ACI 原则澄清**：ACI 的核心不是"减少工具数量"（SWE-agent 只有 ~8 个），而是**每个工具的输出和行为都为 LLM 的认知特点设计**：

| ACI 原则 | SWE-agent 的实现 | Struct Agent 的现状 |
|---|---|---|
| **有界输出** | `view` 限 100 行，需 `scroll` 查看更多 | `file_read` 返回全文，大文件直接溢出 |
| **显式状态** | 维护当前文件 + 行号（cursor） | 无状态跟踪，每次操作需重新指定路径 |
| **自动验证** | 编辑后自动 lint，失败回滚 | 编辑即完成，无后续验证 |
| **信息密度** | 每条输出精心裁剪，去噪 | `shell_exec` 可能返回数千行 stdout |

一个设计良好的 Agent OS 可以有 50 个工具——只要每个都遵循 ACI 原则。Struct Agent 的 20 个工具的问题是**抽象层级太低**（`file_read` = `fs.readFile`，`file_list` = `ls`），没有做任何 LLM 友好的适配。正确的方向不是"减少工具"，而是"**提升每个工具的 LLM 友好度**"。

### 1.7 五层记忆系统过度理论化，实际效用存疑

Struct Agent 的五层记忆（instant/working/episodic/project/semantic）在 README 中被描述为一个完整系统。五层分层在理论上是合理的（对应认知科学中的记忆模型），但实现与理论的差距巨大：

- **实现只有两个工具**：`remember` 和 `recall`，对应五层中的 project 层，其他四层无实际工具/API 可操作。
- **MemoryFusion 是字符串拼接**：将记忆条目拼成文本注入 prompt，没有任何结构化检索或语义排序。
- **层间流转不可见**：理论上 instant→working→episodic 应该自动流转，但代码中没有对应逻辑。
- **对标差距**：Aider 的 RepoMap 没有任何分层理论，但实际效用极高——因为它用 tree-sitter + PageRank 在每次 LLM 调用前自动注入最相关的代码符号。实用主义的"好用"胜过理论完备的"好看"。

**建议**：保留 project 层（手动 remember）作为用户显式知识的存储，其余层合并为一个**统一的自动经验系统**（见 §2.4），由框架自动提取和管理。

### 1.8 Plugin 系统是正确方向，但被低估了

Struct Agent 有 PluginManager + Pipeline（洋葱模型中间件），这是一个正确的架构决策。但它目前只作为"工具注册/拦截"的机制使用。

**Plugin 系统在终极架构中应该成为 Agent OS 的扩展点**——不是给 LLM 用的，而是给开发者用的：

- **post-edit hook**：编辑文件后自动运行 lint/类型检查/格式化（注入为插件）
- **pre-commit hook**：每次 checkpoint 前自动运行相关测试（注入为插件）
- **经验提取器**：每次成功修复后自动提取模式到踩坑库（注入为插件）
- **自定义验证**：项目特定的验证逻辑（如"所有 API 路由必须有 Swagger 注释"）通过插件注入

Plugin 系统让 Agent 框架从"封闭的完整系统"变为"开放的可扩展平台"——这是 Struct Agent 设计中最被低估的部分。

### 1.9 TypeScript 技术选型：双刃剑

Struct Agent 是少数用 TypeScript 写的 coding agent 框架（大部分竞品用 Python）。这个选型的影响：

| | 优势 | 劣势 |
|---|---|---|
| **生态** | 与前端/全栈天然兼容（LSP for TS、Electron 桌面端、npm 生态） | AI/ML 生态薄弱（tree-sitter wasm 可用但不如 Python 原生绑定方便） |
| **类型安全** | 编译期类型检查减少运行时错误 | 对 LLM 输出（JSON/字符串）的类型验证仍需运行时处理 |
| **训练管线** | — | 微调/RL 工具链几乎不在 Node 生态 |
| **部署** | 单文件二进制（esbuild/bun）、轻量 | — |

**对终极架构的影响**：如果终极架构包含 self-play 训练（需要模型微调），TypeScript 不是最优选择——训练管线大概率需要用 Python 做。合理的分工是：**TypeScript 负责 agent 运行时 + Agent OS，Python 负责离线训练管线**。两者通过文件（轨迹数据 JSONL）通信。

### 1.10 Self-Play 训练的适用场景限定

§2.3 提出的 self-play 训练流程（`structfocus-agent self-play --rounds 1000`）是正确方向，但需要精确限定适用场景：

- **不适用于个人小项目**：1000 轮 self-play = ~2000 次 API 调用，成本和收益不成正比。小项目的代码复杂度不足以产生有意义的学习信号。
- **适用于大型项目维护团队**：值得投入计算资源换取长期能力提升。每次 PR 合并后自动跑 self-play 更新项目经验。
- **适用于 Agent 框架自身**：dogfooding——用自己的框架改进自己的框架，产生最相关的训练数据。
- **随机 bug 注入的局限性**：随机注入 bug 可能产生大量无意义的训练样本。需要 bug 注入策略（如：基于 git diff 的最近修改区域注入、基于 lint 规则的反模式注入）才能产生高质量训练数据。

这不是"每个开发者都需要"的功能。它是面向**专业团队和框架开发者**的高级特性。

## 2. 更进一步：终极架构推演

如果 Struct Agent 的目标是"**彻底放飞 AI 的开发能力**"，它需要的不是修修补补，而是一次**架构范式转换**。以下是基于 2025-2026 年前沿研究推演的终极设计。

### 2.1 核心范式转换：从"工具执行框架"到"自进化开发环境"

| 维度 | 当前 Struct Agent | 终极架构 |
|---|---|---|
| **定位** | LLM 的工具执行壳 | LLM 的自进化训练环境 |
| **核心循环** | 用户指令 → 工具调用 → 返回结果 | 代码生成 → 自动测试 → 失败分析 → 自我修正 → 经验沉淀 → 能力提升 |
| **反馈信号** | 工具执行结果（成功/失败） | **可验证奖励**（编译通过、测试通过、性能达标、安全扫描） |
| **记忆** | 对话历史 + 手动 remember | **经验蒸馏**：每次成功/失败自动提取为可复用知识 |
| **上下文** | 静态 I/D 分离 | **动态注意力**：基于当前推理阶段自动选择最相关上下文 |
| **规划** | 五阶段提示词引导 | **自主探索**：模型自己决定何时 explore/plan/execute/verify |
| **安全** | 权限矩阵 + 审批队列 | **执行隔离 + 行为边界**：允许任何操作但限制在沙箱内，事后审计 |

### 2.2 核心机制一：RLVR 闭环 —— 让代码质量自己说话

这是整个架构的基石。**代码开发的独特性在于：它是可验证的**。

```
┌─────────────────────────────────────────────────────┐
│                  RLVR 自进化闭环                      │
│                                                       │
│   生成代码 ──→ 编译 ──→ 运行测试 ──→ 验证通过？       │
│      ↑                                    │          │
│      │                           ┌────────┴────────┐ │
│      │                           │ YES           NO │ │
│      │                           │ 记录成功路径    │ │ │
│      │                           │ 沉淀为经验      │ │ │
│      │                           │                 │ │ │
│      │                           │ 分析失败原因    │ │ │
│      │                           │ 生成修复方案    │ │ │
│      │                           │ 记录失败模式    │ │ │
│      │                           └────────┬────────┘ │
│      └────────────────────────────────────┘          │
└─────────────────────────────────────────────────────┘
```

**与传统 agent 的本质区别**：
- 传统 agent：模型改完代码 → 告诉用户"改好了" → 用户跑测试发现有问题 → 用户反馈 → 模型再改
- RLVR agent：模型改完代码 → **自动验证** → 验证不过 → **自动分析失败原因** → **自动修复** → 循环直到验证通过 → 只把最终结果告诉用户

**可行性分层（重要限定）**：

RLVR 闭环并非在所有层面都能纯框架实现。必须区分两个层次：

| 层次 | 验证类型 | 实现方式 | 可行性 |
|---|---|---|---|
| **框架层可实现** | 编译错误、类型错误、lint 错误 | 确定性检测 + 自动触发修复 | ✅ 完全可行 |
| **需要 LLM 配合** | 测试失败分析、测试生成 | 需要理解业务逻辑 | ⚠️ 依赖 LLM 能力 |

框架层能做到的是"**编译/类型/lint 的自动修复循环**"——这些都是确定性的，不需要理解业务。但"测试驱动的修复循环"需要已有测试存在；"自动生成新测试"是 LLM 层面的能力，不是框架层面的能力。ReVeal（ICLR 2026）解决这个问题的方案是 RL 训练——这不是纯框架层能做的。

**需要新增的基础设施**：
- **编译/类型/lint 自动修复器**：编辑后自动运行，失败则触发修复循环（框架层可实现）
- **失败模式分类器**：将编译错误、测试失败、类型错误自动分类，匹配修复策略
- **经验蒸馏器**：每次 fix 成功后自动提取"原因→修复"模式，写入长期记忆
- **回归守护**：每次修改后自动跑全量相关测试，防止引入新 bug（需要已有测试）

### 2.3 核心机制二：Self-Play 训练 —— AI 给自己出题

Meta FAIR 的 SSR（Self-play SWE-RL）和 DeepSeek 的 AZR（Absolute Zero RL）证明了一个革命性的事实：**AI 可以通过自己给自己出题来变强，不需要人类标注数据**。

Struct Agent 目前完全依赖用户输入作为任务源。终极架构应该内置 self-play 能力：

```
┌──────────────────────────────────────────┐
│           Self-Play 双角色循环             │
│                                           │
│   出题者（Bug Injector）                   │
│   ├─ 分析项目代码结构                      │
│   ├─ 注入有意义的 Bug（非随机破坏）          │
│   ├─ 生成验证该 Bug 的测试                 │
│   └─ 确保 Bug 可被复现                     │
│               ↓                           │
│   解题者（Bug Solver）                     │
│   ├─ 接收 Bug 描述 + 失败测试               │
│   ├─ 定位 + 修复                          │
│   ├─ 运行测试验证修复                      │
│   └─ 成功 → 奖励 / 失败 → 分析             │
│               ↓                           │
│   经验反馈                                 │
│   └─ 成功修复 → 沉淀修复模式               │
│       Bug 太简单 → 出题者提高难度           │
│       Bug 太难 → 出题者降低难度             │
│       形成动态难度课程                      │
└──────────────────────────────────────────┘
```

**这不需要在运行时实时做**——可以作为**离线训练/微调流程**：
1. 用户在项目中运行 `structfocus-agent self-play --rounds 1000`
2. Agent 自动注入 Bug → 修复 Bug → 记录成功/失败轨迹
3. 轨迹数据用于微调模型，提升其在该项目中的代码理解和修复能力
4. 微调后的模型回到运行时使用，能力提升

**关键**：这个流程**完全不需要人类参与**。它是真正的"AI 自己变强"。

### 2.4 核心机制三：经验蒸馏型记忆 —— 不是"记住对话"，而是"学会技能"

当前 Struct Agent 的记忆系统（remember/recall/fusion）本质是"对话存档 + 向量检索"——记住"用户说了什么""我做了什么"。

终极架构的记忆应该是**技能导向的**：

| 记忆类型 | 当前实现 | 终极设计 |
|---|---|---|
| **项目记忆** | remember("用 pnpm 而非 npm") | 自动提取：`{ packageManager: "pnpm", reason: "workspace 支持" }` → 下次 `npm install` 时自动纠正 |
| **踩坑记忆** | remember("tsconfig paths 不生效需要检查 baseUrl") | 自动提取：`{ pattern: "tsconfig paths", symptom: "模块找不到", fix: "检查 baseUrl 与 paths 的 base 是否一致" }` → 遇到类似错误自动提示 |
| **模式记忆** | 无 | 自动提取：`{ pattern: "React useEffect cleanup", common_bug: "忘记 return cleanup 函数", fix_template: "..." }` → 代码审查时自动检查 |
| **修复记忆** | 无 | 每次 RLVR 循环中的成功修复自动记录为 `{ error_signature → fix_strategy }` 映射 |

实现方式：
- **自动提取**（非手动 remember）：每次成功修复或遇到错误后，用一个小模型自动提取关键模式
- **冲突检测**：新记忆与旧记忆矛盾时，自动标记并提示人工裁决
- **衰减 + 强化**：每次被验证有效的记忆增加权重，长期未使用的记忆自动归档

### 2.5 核心机制四：动态上下文 —— 不是"分层缓存"，而是"注意力驱动"

抛弃静态的 I/D 分离。改为：

```
上下文 = 系统基础层 + 注意力层（动态注入）

注意力层的内容由框架自动决定，基于：
├─ 当前推理阶段（需要探索还是需要修改？）
├─ 相关文件（Aider RepoMap 式的 PageRank 排序）
├─ 活跃错误（最近一次编译/测试失败的具体信息）
├─ 相关记忆（经验蒸馏中匹配当前情境的模式）
├─ 关键决策（最近 N 步中做出的不可逆选择）
└─ token 预算（当前窗口剩余量）
```

**框架层自动管理，不需要模型调用 focus/forget**。模型的 tool list 里甚至不应该有 focus/forget——这些是框架的内部决策。

**动态注入规则**：
- token 使用率 > 70%：自动触发压缩（LLM 摘要，非字符串截断）
- 连续 3 次工具调用未修改文件：自动注入"你似乎卡住了，考虑换一个方法"
- 测试失败：自动注入失败的具体 diff + 相关代码片段 + 历史中类似错误的修复方式
- 发现新文件：自动将该文件的关键符号注入上下文（Aider RepoMap 模式）

### 2.6 核心机制五：Agent OS —— 从"工具集"到"开发环境"

当前 Struct Agent 给 LLM 的是"20 个工具"。终极架构应该给 LLM 一个**完整的开发环境**：

```
Agent OS（给 LLM 的抽象操作系统）
├── 文件系统
│   ├── 项目文件（可读写，受信任根限制）
│   ├── 依赖包（只读，node_modules 等）
│   └── Agent 工作区（临时文件、中间产物）
│
├── 进程管理
│   ├── 运行命令（编译、测试、lint）
│   ├── 后台进程（dev server、watch mode）
│   └── 进程间通信（pipe 数据流）
│
├── 代码智能
│   ├── 符号索引（tree-sitter，所有语言）
│   ├── 类型信息（LSP，实时）
│   ├── 依赖图（文件间 import 关系）
│   └── 调用图（函数调用链）
│
├── 验证系统
│   ├── 自动测试生成（为新代码生成测试）
│   ├── 回归测试运行（全量或受影响范围）
│   ├── 类型检查（TypeScript/Python mypy 等）
│   └── lint 检查（ESLint/Ruff 等）
│
├── 版本控制
│   ├── 自动 checkpoint（每次成功修改后）
│   ├── 回滚（checkpoint 之间自由跳转）
│   └── 分支（探索性修改不污染主线）
│
└── 经验系统
    ├── 项目记忆（自动提取的项目知识）
    ├── 踩坑库（错误 → 修复映射）
    └── 模式库（常见代码模式 + 最佳实践）
```

**关键设计决策**：LLM 不需要知道这个 Agent OS 的每个子系统。它只需要知道：
- "我能读写文件"
- "我能运行命令"
- "我能看到代码结构"
- "改完代码会自动验证"

其余全部是框架层自动做的事情。**好的 Agent 框架不应该让模型操心基础设施**。

## 4. 能否让 AI 开发进入新时代？

**直接回答：Struct Agent 的当前设计不能。但它的骨架 + 本报告的推演方向，可以。**

### 4.1 当前设计的根本天花板

Struct Agent 把 LLM 当成一个"需要被管理、被限制、被引导"的外部实体——用工具定义约束行为，用权限矩阵限制操作，用五阶段提示词控制节奏。这是过度偏向 **"监工模式"**。

而真正能"放飞 AI 开发能力"的架构应该把 LLM 当成一个**能在受控环境中自主探索、试错、学习的智能体**——给它环境、给它反馈、让它自己变强。

**但这不是简单的"监工 vs 教练"二选一**。真实系统中，两者必须**分层共存**：

```
人类确认层（不可绕过）
├── git push to main
├── 删除生产数据
└── 发布到 npm / deploy
        ↑
    需要人类审批
        ↑
══════════════════════════════════
        ↑
    沙箱内完全自由
├── 读写项目文件
├── 运行任意命令
├── 修改配置
└── 创建/删除分支
        ↑
    事后可审计，但执行时不阻止
```

Struct Agent 的问题不是"有权限矩阵"——权限矩阵的**意图**是对的，且经复核**矩阵确实可达**（`harness.ts` 的 `toolToOperation()` 把 `file_delete→"delete"`、`git_push→"git-push"` 接到 `Policy.checkPermission()`）。问题是**过度偏向监工端**：
- 权限映射是手工前缀表（`toolToOperation`），新增工具易静默绕过检查——见 CODE_REVIEW 缺陷 F
- 五阶段提示词强约束推理节奏
- 六原语（实为 5 个：focus/forget/reflect/remember/recall）把框架层应该主动做的事情推给模型

**修正后的架构哲学**：终极架构应该是"**沙箱内完全自由 + 关键操作人类确认 + 所有操作可审计**"——监工和教练的分层共存，而非非此即彼。

### 4.2 "新时代"的标志

> **免责声明**：以下标准具有主观性。"新时代"对不同角色的含义不同——对个人开发者可能是"写完需求就能交付"，对团队可能是"维护成本降低 80%"，对研究者可能是"agent 能自我进化"。以下标准是基于"通用 coding agent 能力边界"的推演，请按自身场景校准。

当以下三个条件同时满足，AI 开发就进入了新时代：

1. **AI 能自主完成一个完整的开发任务**（从理解需求到通过全部测试），不需要人类中途介入。
   - 当前：大部分 agent 在 SWE-bench 上 70-80%，但需要精心构造的 prompt 和多次重试。
   - 目标：一次给出自然语言需求，Agent 自主完成，交付通过全部测试的 PR。

2. **AI 能从每次失败中学习**（RLVR + 经验蒸馏），在同一个项目中越用越强。
   - 当前：每次运行都是从零开始的，不会积累经验。
   - 目标：第二次做类似修改时，速度提升 50%，错误率降低 80%。

3. **AI 能自我进化**（self-play），不需要人类持续提供训练数据。
   - 当前：所有 agent 都依赖底层 LLM 的能力，LLM 不更新，agent 不进步。
   - 目标：Agent 在自己的使用过程中积累轨迹数据，定期微调，能力螺旋上升。

### 4.3 实现路径

这不是空中楼阁。以下是可执行的路径：

**Phase 1：从"过度监工"到"验证闭环"（1-3 个月）**
- 在 `Harness.exec` 中加入 post-execution 验证步骤：编辑后自动跑 lint、类型检查、编译验证
- 实现"编译/类型/lint 失败 → 自动分析 → 自动修复 → 再次验证"的框架层 RLVR 内循环
- 不需要改 LLM，不需要训练——**纯框架层面可实现**
- 注意：测试驱动的修复循环需要已有测试，自动测试生成需要 LLM 配合——不在 Phase 1 范围内

**Phase 2：从"对话记忆"到"经验蒸馏"（3-6 个月）**
- 每次 RLVR 循环中的成功修复自动提取为 `{error_signature → fix_strategy}` 映射
- 构建项目级"踩坑库"，在遇到类似错误时自动注入历史修复方案
- 实现记忆的自动衰减 + 强化机制

**Phase 3：从"纯推理"到"Self-Play 训练"（6-12 个月）**
- 实现离线 self-play 流程：Agent 在项目代码库中自主注入 Bug → 修复 Bug → 收集轨迹
- 轨迹数据用于微调模型（或用 RLVR reward 做在线 RL）
- 实现"越用越强"的正反馈循环

## 5. Struct Agent 的设计中，什么应该保留？

在推倒重来的冲动中，有一些设计是真正有价值的，应该作为终极架构的基因保留：

| 保留项 | 原因 |
|---|---|
| **Git 版本化的 checkpoint 机制** | 虽然 D-Context 的 Git 版本化是过度设计，但"每次修改后自动 checkpoint + 可回滚"是极好的。终极架构中应该保留为 Agent OS 的版本控制层 |
| **2PC 原子事务** | 多文件修改的原子性保证是正确且必要的。终极架构中保留为文件系统层的基础保证 |
| **结果脱敏（sanitizeOutput）** | 敏感信息过滤是安全底线。终极架构中保留为输出层的基础过滤 |
| **EventBus + PluginManager** | 可扩展的插件体系是好的。终极架构中保留为 Agent OS 的扩展点 |
| **Session Resume** | 跨会话恢复是实用功能。终极架构中保留为编排层的持久化能力 |
| **代码符号提取（code_symbols）** | 方向正确。终极架构中升级为完整的 tree-sitter + PageRank 符号索引 |
| **PluginManager + Pipeline** | 洋葱模型中间件是极好的可扩展架构。终极架构中升级为 Agent OS 的扩展点（post-edit hook、pre-commit hook、自定义验证等），让框架从"封闭系统"变为"开放平台" |
| **TypeScript + Monorepo 工程化** | 类型安全 + workspace 组织是扎实的工程基础。终极架构中保留为 agent 运行时的技术栈，离线训练管线用 Python 作为补充 |

## 6. 总结：从"Struct"到"Evolve"

Struct Agent 的名字暗示了它的核心哲学：**结构化、确定性、受控**。

但真正能"放飞 AI 开发能力"的系统，应该叫 **Evolve Agent**——**进化、自主、生长**。

**核心转变**：
- 从"框架控制模型"到"环境赋能模型"
- 从"人类主导的对话"到"AI 自主的探索"
- 从"每次从零开始"到"越用越强"
- 从"权限闸门"到"沙箱自由 + 事后审计"

这不是对 Struct Agent 的否定。相反，Struct Agent 为这个终极架构奠定了很好的基础——它证明了 TypeScript monorepo 下可以构建完整的 agent 框架，它的 2PC 事务、EventBus、插件体系都是扎实的工程。**它只是选错了顶层哲学**——把 LLM 当成了需要被管理的工具，而非需要被赋能的智能体。

**下一步**：如果你认同这个方向，不需要从零重写。在现有 Struct Agent 的基础上，按 Phase 1 → Phase 2 → Phase 3 逐步演进——每一步都是增量改进，每一步都让 Agent 变得更"自主"。当三个阶段走完，Struct Agent 自然就变成了 Evolve Agent。
