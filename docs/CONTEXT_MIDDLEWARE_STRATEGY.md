# StructFocus 上下文引擎：上下限高发展路线

> 版本：v0.1.0-draft ｜ 日期：2026-07-15
> 主线锚定：StructFocus 的核心资产是**上下文引擎**（哈佛 I/D 分离 + 六原语 + 预算桶 + 价值驱逐），差异化定位是**「注意力聚焦」而非「压缩裁剪」**。本文件回答：要不要做完整 Agent、如何做中间层、闭源环境怎么破、上下限高的发展路线怎么走。

---

## 0. 事实校准（先纠偏，避免路线基于错误前提）

历代审查中有一条判断需修正，防止路线跑偏：

- **旧判断（错误）**：「六原语只落了 3 个，remember/recall 没实现。」
- **核实后（正确）**：`structfocus-agent.ts` L120 声明 `CONTEXT_TOOLS = {focus, forget, reflect, remember, recall}` 五个工具**全部已接线**——`focusFile/forgetFile/reflect` 在 `context/src/manager.ts`（L695/L743/L757），`remember/recall` 在 `agent` 层的 `memory` 模块（`structfocus-agent.ts` L1044-1062，`record` / `searchHybrid` / `searchSync`）。代码注释称「六原语」是命名遗留，实际为 5 个工具。
- **真正的 gap（本轮结论）**：不是「没实现」，而是以下三点：
  1. **注意力是模型手动工具，框架从不主动接管**。`manage()`（L831-849）默认走 `compressOldEntries`（压缩、损信息）→ `evictLowValue` → `truncate`。注意力设计被边缘化为 opt-in，引擎在干你想超越的「压缩」。
  2. **驱逐不看任务**。`evictionScore`（L1133）只看人工重要性 / 访问频率 / 时间衰减 / 体积，**完全不知道「当前子任务在做什么」**，无法按任务承载力淘汰。
  3. **无注意力审计 / 浪费度量**。`reflect()` 只报 token 数与预算%，无法量化「窗口里多少 token 从未被引用或促成下一步」。

> 结论：你离「零浪费注意力」的距离，是「机制在但没被激活 + 没接任务状态 + 没度量」，不是「从零开始」。**这比原先判断乐观——工程量比想象小。**

## 0.5 价值主张（双支柱）：优雅上下文 + 更高正确率

用户定调：StructFocus 的卖点不是「又一个上下文库」，而是**两个并列的、可对外讲的承诺**：

| 支柱 | 一句话卖点 | 我们的差异化机制 | 路线落点 |
|---|---|---|---|
| **① 优雅上下文** | 有限的注意力窗口被分配到**核心**，而非被动删料 | 「注意力聚焦」而非「压缩裁剪」：I/D 分离 + 六原语 + 预算桶 + evictionScore 价值驱逐，按**任务承载力**保留（信号密度高），压缩则损信息 | Phase 1 框架接管六原语 + 任务相关性驱逐 + 注意力审计 |
| **② 更高正确率** | 错的代码在进上下文 / 进用户仓库前就被拦下修复 | 编辑后框架层自动跑 tsc/lint/test（RLVR 内循环），不通过不提交；验证结果作为 **observation 写回 D-Context**，让上下文携带「已验证正确」的信号 | PDR M3 验证闭环 + M4 结构化事件流 |

**两支柱为何不违背「主线是上下文」的纪律**：
- 支柱① 是上下文的**分配方式**（信号密度）。
- 支柱② 的验证产出**本身就是上下文的一类高质量条目**——「哪些改动通过了编译/测试」是模型后续决策最该携带的信号。验证闭环不是脱离上下文的另起炉灶，而是「让上下文携带已验证信号」。两者都落在同一套上下文引擎与标准 API 上（CONTEXT_API_DESIGN.md）。

**两支柱的协同效应**：
- 支柱① 让模型**看对东西**（信号密度高）；
- 支柱② 让模型**做对事情**（错的被拦下，信号带可信度标签）。
- 单有①，模型可能看得很准但产出仍可能错；单有②，模型在噪声上下文里验证也白费。两者叠加 = 「更聪明也更可靠」的 Agent 底座。

**这一表述直接解决了「只做上下文=卖原料」的担忧**：
第十四轮指出「仅上下文管理 = 卖原料非产品，天花板低」。双支柱把卖点从「更好的记忆库」**升级为「更对、更省心的 Agent 表现」**——这是产品级 outcome，不是原料级特性。它同时撑起下限（企业愿为「不出错 + 不浪费」付费）与上限（可被实证、可成「层」）。

## 1. 战略定调：做「上下文中间层」，不做完整体 Agent

### 决策
**不做完整体 Agent 产品，做所有 Agent 的「上下文中间层（Context Layer）」。**

### 为什么不做完整 Agent（三条硬理由）
| 理由 | 说明 |
|---|---|
| **正面战打不过** | OpenSquilla 已实现路由/compaction/记忆/自进化，Apache-2.0 开源、5.5k stars、估值 1 亿。你做同形态产品，胜算低、且慢一拍。 |
| **工程量巨大且是闭源壳** | 完整 Agent 需要多模型路由、沙箱、多通道（飞书/钉钉）、技能市场——这些本质是对闭源 LLM API 的封装，护城河浅、易被平台吸收。 |
| **平台吞噬风险** | 大模型厂有强烈动机**原生做上下文管理**（省的是它们自己的推理算力）。你做完整 Agent，越做越像给平台铺路。 |

### 中间层定义
```
┌─────────────────────────────────────────────┐
│  Agent 框架 / Builder（LangChain、LlamaIndex、 │
│  AutoGen、CrewAI、OpenSquilla、自研……）        │
└───────────────┬─────────────────────────────┘
                │  调用标准 Context API
┌───────────────▼─────────────────────────────┐
│  StructFocus Context Middleware（你的层）      │
│  · 注意力聚焦（focus/forget 框架接管）         │
│  · 价值驱逐（接任务状态）                      │
│  · 注意力审计（浪费度量）                      │
│  · 跨模型一致上下文策略                        │
└───────────────┬─────────────────────────────┘
                │  消费模型输入输出（模型是黑盒）
┌───────────────▼─────────────────────────────┐
│  闭源 LLM API（OpenAI / Claude / GLM …）       │
└─────────────────────────────────────────────┘
```
**你的层在模型外面、Agent 框架下面**——不绑任何模型、不碰任何 Agent 产品逻辑。

## 2. 核心问题逐答

### 2.1 还要不要做一整个 Agent？
**不要。** 完整 Agent 是「产品」，变现靠用户规模与网络效应，正面竞争 OpenSquilla 且没有先发优势。把同一份引擎做成「层」，边际成本极低（卖一份 License 和卖一万份，引擎还是那套），且避开了平台吞噬——你做层，平台做产品，它们是**上下游关系**而非竞争关系。

### 2.2 做中间层怎么做？
三个具体动作：
1. **定义标准 Context API**（见 §4 Phase 2 交付物）：`loadContext(taskState)` / `commitResult(result)` / `focus(scope, strategy)` / `evict(score, taskRelevance)` / `audit()`。任何 Agent 框架接这几行就能用。
2. **写适配器（Adapter）**：为 LangChain Memory、LlamaIndex、OpenSquilla 等各写一个 adapter，让它们用你的引擎**替代原生记忆/压缩**。这是「成为层」的关键——接入面越广，标准话语权越强。
3. **模型无关**：只消费模型输入输出，通过标准接口（OpenAI 格式 / 任意 provider）接 LLM。闭源模型直接可用，无需知道内部。

### 2.3 闭源环境怎么破？
| 闭源物 | 对你的影响 | 应对 |
|---|---|---|
| **LLM API 闭源**（OpenAI/Claude/GLM） | **无影响，反是机会** | 上下文引擎在模型外工作，只消费 I/O。Builder 跨模型需要「一致的上下文治理」，你的层提供跨模型统一策略——闭源反而放大你的价值。 |
| **IDE 闭源**（VS Code/JetBrains） | 只影响分发 | 做插件/扩展，不碰 IDE 内核；IDE 只是你的一个入口，不是依赖。 |
| **模型厂原生上下文** | 威胁但可规避 | 护城河 = 「模型无关 + 注意力聚焦深度 + 垂直代码场景」。模型厂做通用压缩，你做代码符号级聚焦——它不会深耕你的垂直。 |
| **评测/基准闭源** | 无影响 | 自有基准（§4 Phase 3）即可，不需要它们的数据。 |

> 核心洞察：**上下文管理是少有的「闭源友好」赛道**——它工作在模型边界外，模型越黑盒、越碎片化，统一上下文治理的需求越强。

## 4. 各 Phase 具体交付物（含代码落点）

### Phase 0 — 引擎基本功补全（下限地基）
**目标**：让引擎默认行为是「不浪费」而非「溢出才减料」。
- **P0-1 自动化 compaction**：`manage()` 接近软限时被动触发结构化压缩（保留用户目标/状态/失败/工具结果），替代当前手动 `compressOldEntries`。
  - 落点：`packages/context/src/manager.ts` L831-849 `manage()` 逻辑重构。
- **P0-2 工具结果预处理**：裁掉噪声（长输出已截断，补 HTML/重复内容过滤）。落点：`harness` 工具结果回写 D-Context 处（接 M4 事件流）。
- **P0-3 cache 感知排布**：稳定前缀（I-Context）放可缓存段，降重复计费。落点：`manager.ts` I-Context 构造 + `dynamic-prompt.ts`。
**验收**：125000 预算下，相同任务峰值 token 下降 15%+，无信息损。

### Phase 1 — 框架接管六原语 + 任务相关性驱逐（让注意力真发生）
**目标**：把「模型手动工具」变「框架自动策略」——这是你差异化落地的核心动作。
- **P1-1 框架接管 focus/forget/reflect**：token > 70% 自动 `evictLowValue` + 关键动作后自动 `reflect`；不再只「提醒模型」（structfocus-agent L535-540 的提醒改为框架直接执行）。落点：`structfocus-agent.ts` L531 `manage()` 调用点 + `CONTEXT_TOOLS` 由工具降级为内部 API。
- **P1-2 补 `remember`/`recall` 的框架触发**：重要决策/约定自动 `remember`，相关任务自动 `recall`（不再等模型自觉）。落点：`memory` 模块调用封装。
- **P1-3 任务相关性驱逐**：`evictionScore`（L1133）接入当前子任务状态（正在编辑的文件/符号/失败测试），按「任务承载力」而非时间淘汰。落点：`manager.ts` L1133 `evictionScore()`。
- **P1-4 删死代码**：`EVICTION_ORDER`（budget.ts L43）已部分生效，复核后移除未接管线，避免误导。
**验收**：不调用任何上下文工具时，引擎仍能按任务聚焦/淘汰；任务成功率对比基线提升。

### Phase 2 — 开源 SDK + 私有化 License（保底收入）
**目标**：B2B 基础设施变现，撑起「下限」。
- **P2-1 开源 Core SDK**（Apache-2.0）：`@structfocus/context` npm 包，含 Phase 0/1 全部能力 + 标准 Context API。
- **P2-2 商业版**：trace 回放、注意力浪费看板、成本分析（类 LangSmith/Sentry）。
- **P2-3 私有化 License**：金融/政企/军工数据不出域，一次性授权 + 年维护费。
**收入性质**：确定、续费黏性强、客单价高。**这是「下限高」的落点。**

### Phase 3 — 基准实验：注意力 vs 压缩（上限子弹）
**目标**：把「我们更有优势」从口号变证据。
- **P3-1 基准设计**：相同 token 预算（如限 125000），A 组压缩裁剪（对标 OpenSquilla 路线），B 组注意力聚焦（你的六原语自动调度），跑同一批代码任务（用 StructAICoding 场景）。
- **P3-2 指标**：任务成功率、token 效率（成功率/千 token）、注意力浪费率（窗口未引用 token 占比）。
- **P3-3 产出**：技术报告 / 论文级资产 / 融资叙事。
**关键**：若 B 组胜出 → 你有「注意力 > 压缩」的实证，这是发论文、做技术品牌、融资的子弹。若平/负 → 回头修正 Phase 1 策略（诚实迭代，不粉饰）。

### Phase 4 — 成为「层」：适配器 + 标准（生态位）
**目标**：让主流框架用你的引擎，卡住「层」的位置。
- **P4-1 适配器**：LangChain Memory Adapter、LlamaIndex Adapter、OpenSquilla Adapter——让它们一键替换原生记忆。
- **P4-2 ContextLayer 标准草案**：定义跨框架的上下文治理接口，推动成为事实标准。
**逻辑**：接得越广，被平台「吸收」的代价越高（你已是生态一部分），且 Builder 迁移成本形成护城河。

### Phase 5 — 注意力审计 SaaS + 垂直代码深耕（赚大）
**目标**：指数收入切口。
- **P5-1 Sentry for Context**：SaaS 告诉 builder「你的 Agent 注意力浪费在哪一格、哪个工具结果占窗最大、哪次 focus 从未被引用」。这是**无任何平台给 builder 的可见性**，真空白、可垄断。
- **P5-2 垂直代码符号级注意力**：tree-sitter 符号索引喂上下文（而非整文件），编辑后验证闭环（M3）结果喂回 D-Context。平台不做深，你做透。落点：接 M2 技能仓库 + M3 验证层。
**收入性质**：网络效应 + 订阅 + 高毛利交付，这是「上限高」的落点。

### Phase 6 — 战略退出
若 ContextLayer 被足够多 Agent 采用 → 战略价值远超营收，被云厂/IDE 厂收购（类似 LangChain/Rasa 路线，但更聚焦上下文这一层）或被采纳为标准。**顺带发生，不强求。**

## 5. 闭源环境具体应对清单
- **LLM API 闭源**：直接接（OpenAI/Claude/GLM 均消费 I/O），无需模型内部权限。你的卖点是「跨模型一致上下文策略」。
- **IDE 闭源**：插件入口，不依赖内核；VS Code/JetBrains 扩展市场分发。
- **模型厂原生上下文**：用「垂直代码场景 + 注意力聚焦深度」错位竞争；不做通用压缩。
- **评测数据闭源**：自有基准（Phase 3）自足。
- **依赖闭源库**：仅用开源运行时（Node/TS），不引闭源上下文相关依赖，保持层的中立性。

## 6. 风险与对冲
| 风险 | 概率 | 对冲 |
|---|---|---|
| 平台原生做上下文，吞噬需求 | 中高 | 模型无关 + 垂直深耕 + 先卡「层」生态位 |
| Phase 3 实验 B 组未胜出 | 中 | 诚实迭代 Phase 1；注意力聚焦可能在特定场景（长任务/代码）仍胜，缩窄战场 |
| 开源 SDK 无人用（冷启动） | 中 | 先做私有化 License（付费客户验证），再叠开源获采用基数 |
| 适配器维护成本随框架迭代上升 | 中 | 只适配 Top 3 框架，标准稳定后由社区补 |
| 团队资源不足，Phase 1 未完工即想做产品 | 高 | 设检查点（§7）：Phase 1 未验收前禁止做完整 Agent |

## 7. 决策检查点（何时该转向做完整 Agent）
**原则：Phase 1 未验收前，绝不启动完整 Agent。**
- ✅ 转向信号：Phase 3 实验证明注意力显著优于压缩 + Phase 4 已有 ≥2 个主流框架接适配器 + 私有化 License 月收入覆盖团队成本 → 此时可考虑用自有引擎做「垂直代码 Agent」作为旗舰示范（不是通用 Agent）。
- ❌ 禁止信号：仅凭「设计更优」就想做产品、或为融资故事提前做壳 → 回到 §1 三条硬理由。

## 9. 一句话总结
**不做完整 Agent，做所有 Agent 下面的「上下文中间层」；下限靠私有化 License + 开源 SDK 兜住确定收入，上限靠「注意力 > 压缩」的实证 + 成为层 + 注意力审计 SaaS 打开指数空间；闭源环境反是机会（工作在模型外、跨模型一致治理）；Phase 1 未验收前禁止做产品。**
