# 上下文优雅降级：频率分层替代应急驱逐

> 日期：2026-07-15
> 触发：用户发现三层应急管理（70/85/90%）是"满了再清"的错误范式
> 状态：设计重构提案（替代 Phase 0-1 的 autoManage）

---

## 1. 旧模型的问题

```
三层应急管理（已舍弃）:
  
  70% → 驱逐 15%         ← 问题: 为什么不一开始就不塞垃圾?
  85% → 驱逐 + 压缩       ← 问题: 为什么等到 85% 才压缩?
  90% → 强制 forget       ← 问题: 这是应急 panic，不是优雅管理
  
  本质: 等油箱报警了才找加油站。
```

## 2. 新模型：哈佛分层 × 频率衰减

### 2.1 核心思想

```
不是 "满了再清"。
而是 "每一条信息都有温度，温度随时间衰减，温度决定了它待在哪个层级"。
```

### 2.2 分层架构

```
┌──────────────────────────────────────────────────┐
│  L0: IMMUTABLE CONTEXT (I-Context)               │
│  ─────────────────────────────────────            │
│  静态，全会话不变。可复用 KV Cache。              │
│                                                  │
│  • System Prompt (角色、能力边界)                 │
│  • Tool Schemas (可用工具及参数)                  │
│  • Rules & Constraints (安全规则、禁止操作)       │
│  • Host Environment (OS, shell, workspace path)   │
│  • Dev Preferences (editor, formatter, linter)    │
│  • User Profile (身份、偏好、上下文)               │
│                                                  │
│  → 永远在上下文。永不压缩。永不驱逐。              │
│  → 约 8-15K tokens                               │
├──────────────────────────────────────────────────┤
│  L1: ACTIVE SET (Hot Cache)                      │
│  ─────────────────────────────────────            │
│  当前工作集。每步可能变化。                        │
│                                                  │
│  • 正在编辑的文件（完整内容 + 符号表）             │
│  • 当前子任务描述                                  │
│  • 最近 1-3 步的错误/异常                         │
│  • 刚刚 focus 的符号/函数                          │
│  • 当前 git diff (已修改但未提交)                  │
│                                                  │
│  → 完整保留。压缩延后到降入 L2 时。                │
│  → 约 20-40K tokens（取决于文件大小）             │
│  → 温度阈值: > 0.7                                │
├──────────────────────────────────────────────────┤
│  L2: RECENT HISTORY (Warm Cache)                 │
│  ─────────────────────────────────────            │
│  最近几轮交互。已结构化压缩。                      │
│                                                  │
│  • 最近 N 步的 [目标] [动作] [发现] [下一步]       │
│  • 已放弃的方案及原因                              │
│  • 刚完成的子任务胶囊                              │
│  • 相关文件摘要（不是全量内容）                    │
│                                                  │
│  → 结构化压缩后保留。可展开。                      │
│  → 约 10-30K tokens (压缩后)                     │
│  → 温度阈值: 0.3 ~ 0.7                            │
├──────────────────────────────────────────────────┤
│  L3: PACKAGED MEMORY (Cold Storage)              │
│  ─────────────────────────────────────            │
│  已完成、已封装、按需召回。                        │
│                                                  │
│  • 已封装的知识胶囊                                │
│  • 已完成子任务的完整上下文指针                     │
│  • 历史决策记录                                    │
│  • 被 L2 驱逐的非活跃文件                          │
│                                                  │
│  → 不占上下文。仅在 recall 时注入。                │
│  → 约 0 tokens（不占窗口）                        │
│  → 温度阈值: < 0.3                                │
└──────────────────────────────────────────────────┘
```

## 3. 温度模型

### 3.1 温度计算公式

```
temperature(t) = baseTemp × frequencyFactor × recencyFactor × relevanceFactor

frequencyFactor:
  每被访问一次 (LLM 引用 / 工具输出) → +0.15 (上限 1.0)
  没有被访问的步数 k → × 0.85^k

recencyFactor:
  过去 1 步: × 1.0
  过去 3 步: × 0.9
  过去 5 步: × 0.7
  过去 10 步: × 0.4
  超过 20 步: × 0.1

relevanceFactor:
  当前编辑文件: 1.0
  同目录文件: 0.7
  关联测试文件: 0.6
  失败测试引用: 0.8
  无关文件: 0.2
  LLM 显式 focus: 锁定 1.0 (直到显式 unfocus)
```

### 3.2 温度 → 层级映射

```
temp > 0.7  → L1 (Hot: 完整保留)
temp 0.3-0.7 → L2 (Warm: 结构化压缩)
temp < 0.3  → L3 (Cold: 打包为胶囊，移出窗口)
```

### 3.3 温度衰减轨迹（示例）

```
Step 0:  read_file(auth.ts)     → temp = 1.0  | L1 完整
Step 1:  edit auth.ts           → temp = 1.0  | L1 (刚才用了)
Step 2:  没用 auth.ts           → temp = 0.85 | L1 (轻微衰减)
Step 4:  没用 auth.ts           → temp = 0.61 | L2 (压缩!)
Step 8:  没用 auth.ts           → temp = 0.32 | L2 下沿
Step 12: 没用 auth.ts           → temp = 0.16 | L3 (打包胶囊)
Step 20: 回忆 auth.ts           → temp = 0.16 + 0.15 = 0.31 | 回到 L2!
Step 21: 继续用 auth.ts         → temp = 0.31 + 0.15 = 0.46 | L2 上升
```

## 4. 每步优雅流程

### 4.1 旧流程（应急）

```
Turn 完成
  → usePercent = 计算
  → if usePercent >= 70%: evict 15%    ← 应急
  → if usePercent >= 85%: compress      ← 更应急
  → if usePercent >= 90%: force forget  ← panic
```

### 4.2 新流程（优雅）

```
Turn 完成
  → 1. 更新温度: 本步引用的条目 +0.15
  → 2. 温度衰减: 未引用的条目 × 0.85
  → 3. 层级迁移:
      temp > 0.7 → L1 (完整)
      0.3 ≤ temp ≤ 0.7 → L2 (压缩)
      temp < 0.3 → L3 (打包胶囊)
  → 4. 自动打包: temp 首次跌破 0.3 → 封装为知识胶囊
  → 5. L3 条目移出上下文窗口
  → 6. 如果窗口仍然超限 (极罕见): 优先压缩 L2 中 temp 最低的条目

每步成本: O(n) 条目数，纯本地计算，< 1ms
```

## 5. 自动打包（知识胶囊）

### 5.1 触发条件

不是"上下文满了才打包"，而是"温度自然跌破阈值就打包"。

```
temp < 0.3 → 自动打包触发器:

  ┌─────────────────────────────────────────┐
  │ 检查: 这个条目（或这组条目）是否构成     │
  │       一个有意义的"工作单元"？           │
  │                                         │
  │ 是 → 打包为知识胶囊                      │
  │ 否 → 等待更多相关条目一起降温再打包      │
  └─────────────────────────────────────────┘
```

### 5.2 工作单元自动检测

```
检测规则:
  1. 同一次子任务的所有相关条目一起降温 → 一起打包
  2. 同一文件的讨论跨越多个 turn → 合并打包
  3. 错误 + 修复 成对出现 → 打包为"bug修复"胶囊
  4. 子任务标记为 complete → 对应所有条目打包

自动打包内容:
  ├── 需求/目标
  ├── 涉及文件及修改
  ├── 关键决策（为什么这样改）
  ├── 放弃的方案
  ├── 测试结果
  └── 已知限制/技术债
```

### 5.3 打包后

```
打包前: 原始上下文 15K tokens (L2 中)
打包后: 胶囊指针 80 tokens (L3)
       "[capsule] fix_token_refresh_race: auth.ts +45/-12, 12 tests pass,
        known: 0.1% edge case race (#452)"

打包后胶囊存储在外部:
  memory/capsules/fix_token_refresh_race_20260715.json
  → 可通过 recall 完整展开 (15K tokens 原样恢复)
```

## 6. 对比

| 维度 | 旧模型 (70/85/90%) | 新模型 (频率分层) |
|------|---------------------|---------------------|
| 触发时机 | 窗口拥挤时 | **每一步** |
| 操作性质 | 应急驱逐 | **自然降温 → 优雅迁移** |
| 延迟 | 可能触发压缩 spike | **平滑，无 spike** |
| 信息保真 | 截断丢尾部数据 | **完整保留，打包到 L3** |
| 可逆性 | 驱逐后不可恢复 | **胶囊可展开，完全可逆** |
| LLM 注意力 | 清除垃圾但时机不佳 | **始终只有温数据在窗口** |
| 内存/CPU | 空闲时不做事 | **每步微计算，< 1ms** |
| 自动打包 | 没有 | **温度跌破 0.3 自动封装胶囊** |

## 7. 窗口用量的预期效果

```
旧模型 (200K 窗口):
  Steps 1-5:   40K  (正常)
  Steps 6-8:   80K  (快速膨胀)
  Steps 9-10:  140K (触发 70% → 应急驱逐)
  Steps 11-12: 90K  (驱逐后回落)
  Steps 13-15: 150K (再次膨胀)
  Steps 16-17: 170K (触发 85% → 应急压缩)
  Steps 18-19: 185K (触发 90% → panic)
  → 锯齿状，每 5-8 步一次 panic

新模型 (200K 窗口):
  Steps 1-5:   35K  (正常)
  Steps 6-10:  42K  (老条目已降入 L2 压缩)
  Steps 11-15: 45K  (更老的已打包降入 L3)
  Steps 16-20: 48K  (稳定)
  Steps 21-25: 50K  (稳定)
  → 平滑曲线，无 panic
  
  每步额外 60-120 token (温度更新开销 ≈ 无)
```

## 8. 与哈佛 I/D 架构的关系

```
L0 = I-Context (指令层，静态)
L1 = D-Context 高频层 (数据层，完整)
L2 = D-Context 中频层 (数据层，压缩)
L3 = D-Context 低频层 (存储层，指针)

I-Context 永远在窗口。D-Context 的 L1+L2 在窗口。L3 在外部。
I-Context 决定 Agent 能做什么 (能力边界)。
D-Context 决定 Agent 当前关注什么 (注意力焦点)。
```

## 9. 实现路径

### 9.1 需要改的

```
ContextManager:
  ├── 删: autoManage() → 不再需要
  ├── 删: evictionScore() → 替换为 temperature()
  ├── 删: 三层阈值常量 (70/85/90%)
  ├── 加: updateTemperatures() → 每步温度更新
  ├── 加: migrateTiers() → L1→L2→L3 迁移
  ├── 加: autoPackage() → 自动封装知识胶囊
  ├── 改: manage() → 保留但降低优先级 (仅超限时降级)
  └── 改: preprocessToolOutput() → 保留

ContextEntry:
  ├── 删: evictionScore
  ├── 删: evicted / evictedAt
  ├── 加: temperature: number (0-1)
  ├── 加: tier: 'L1' | 'L2' | 'L3'
  ├── 加: accessCount: number
  ├── 加: lastAccessedAt: number
  ├── 加: capsuleId?: string
  └── 加: enterTierAt[L1/L2/L3]: number (时间戳)
```

### 9.2 工作量

| 任务 | 工作量 | 说明 |
|------|--------|------|
| 温度模型实现 | 1 天 | temperature() + 衰减 + 迁移逻辑 |
| autoPackage 实现 | 1 天 | 工作单元检测 + 胶囊生成 |
| 旧代码清理 | 0.5 天 | 删 evictionScore / 三层阈值 |

### 9.3 迁移风险

- 已有 Phase 0-1 代码 = 保留 L2/L3 的基础设施（structuredCompress, recall, 胶囊存储）
- 仅替换决策逻辑：从"驱逐评分" → "温度模型"
- 测试套件可复用（17 文件 / 75 tests）
- 回退路径：保留 autoManage 作为 fallback（温度模型为首选，autoManage 兜底超限情况）
