# 自主质询驱动上下文架构 v2

> 日期：2026-07-15
> 触发：用户指出温度分层设计的三个致命问题，
>       并要求解决 LLM 懒惰/跳过/猜错的根本矛盾
> 状态：设计 v2，替代温度分层草案

---

## 0. 三个诚实回答

### Q1: Warm 和 Cold 有什么区别？

**区别只在一件事上：Warm 在窗口里，Cold 在窗口外。**

- L2 Warm: 压缩后的内容在上下文中，LLM 能读到（哪怕只是摘要）
- L3 Cold: 只有一个 80-token 指针在上下文中，LLM **看不见**完整内容

所以 Cold = 盲区。如果 LLM 不主动 `recall:context`，它就是在猜。

**这意味着 L3 不是一个"更省"的机制，它是一个"关门"的操作。关门前必须确认门里没人。**

### Q2: 真的能让 LLM 更省更准吗？

不会自己变省变准。优雅的分层架构不会自动产生正确的结果。**优雅只是好看，好看不保证能用。**

温度模型解决的是"系统不 panic"的问题，不是"LLM 不犯错"的问题。换了一个不会报警的油箱，车还是可能开到沟里。

### Q3: LLM 嫌麻烦跳过怎么办？

**会。一定会的。**

Letta 的 48-74% 准确率已经证明了：LLM 自我管理记忆不可靠。它会：
- 看到一个胶囊指针 → "我大概知道是什么" → 跳过 → 猜错细节
- 上下文拥挤但还能跑 → "懒得管了，赌一把" → 漏关键约束
- 文件改了但没 recall 相关胶囊 → "应该没影响" → 引入回归 bug

**温度分层把这个问题的严重性提高了，因为 L3 的信息完全不可见。**

## 1. 所以真正的矛盾是什么

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│  我们要省 token (减少噪音)                           │
│          ↕                                          │
│  但不能丢准确率 (不能让 LLM 在看不见的地方猜)         │
│          ↕                                          │
│  LLM 不会主动要求看它不知道自己不知道的东西            │
│                                                     │
│  这是三个互相冲突的目标。                             │
│  优雅的分层架构只是把它们排列整齐了，一个都没解决。     │
│                                                     │
└─────────────────────────────────────────────────────┘
```

## 2. 解决方案：推挽双轨

### 2.1 两条线

```
轨道 A: 可逆压缩（推）          轨道 B: 上下文指针（拉）
─────────────────────          ─────────────────────
框架决定 LLM 需要什么           LLM 决定自己还需要什么
框架主动推入上下文              LLM 主动拉取
不依赖模型自觉                  不依赖模型自觉
关键是"什么时候推"             关键是"什么时候拉"

现在的问题是:
  轨道 A 没有。全丢了或截断了。
  轨道 B 不可靠。LLM 懒得拉。

需要:
  轨道 A 必须存在。框架推。
  轨道 B 必须有强制拉取的触发条件。不让 LLM 偷懒。
```

### 2.2 核心洞察

> **LLM 不需要"所有信息"。但 LLM 不知道自己需要什么信息的边界在哪。**

所以设计原则变成：

> **把 LLM 的"我需要知道"边界从 LLM 脑子里移出来，放进框架的判断逻辑里。**

## 3. 推挽架构

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│  L0: META-CONTEXT (永远在)                          │
│  ─────────────────────────                          │
│  静态。不变。框架定义 Agent 的能力边界。              │
│  • System Prompt • Tool Schema • Rules • Host Env •  │
│    Dev Preferences • User Profile                    │
│  → 8-15K tokens                                     │
│                                                     │
├─────────────────────────────────────────────────────┤
│                                                     │
│  L1: ACTIVE WINDOW (热数据，完整在)                  │
│  ─────────────────────────────────                  │
│  每步自动填充。框架判断 LLM 此时需要什么。推入。      │
│                                                     │
│  推入规则 (框架决定, 不依赖 LLM 自觉):               │
│  ┌─────────────────────────────────────────────┐    │
│  │ ① 当前编辑文件 (焦点文件 → 完整内容)         │    │
│  │ ② 当前子任务定义 (Task Spec)                 │    │
│  │ ③ 最近 3 步的工具结果 (去噪后)               │    │
│  │ ④ 当前 git diff (已改未提交)                 │    │
│  │ ⑤ 活跃文件关联的胶囊摘要 (如果胶囊包含      │    │
│  │    未解决的 TODO/KNOWN_BUG/EDGE_CASE,        │    │
│  │    → 必推, 不允许跳过)                       │    │
│  │ ⑥ 最近 5 步内产生的错误/异常 (必推)          │    │
│  └─────────────────────────────────────────────┘    │
│  → 20-50K tokens (取决于文件大小)                   │
│                                                     │
├─────────────────────────────────────────────────────┤
│                                                     │
│  L2: COMPRESSED HISTORY (温数据，压缩后在)           │
│  ───────────────────────────────────────             │
│  结构化压缩后的对话历史。                            │
│                                                     │
│  压缩格式:                                           │
│  ┌─────────────────────────────────────────────┐    │
│  │ [Step 8] 探索 auth.ts 双 token 刷新方案      │    │
│  │   动作: read(auth.ts), read(auth.test.ts)    │    │
│  │   发现: 第 87 行缺少 mutex 锁                 │    │
│  │   决定: 使用 mutex 而非 lockfile              │    │
│  │   原因: lockfile 在并发场景下性能差 3x        │    │
│  │   放弃: 全局锁方案 (耦合太强)                 │    │
│  │   → 📦 capsule:fix_token_refresh (完整上下文)  │    │
│  │      [若有已知限制, 下文会自动注入警报]        │    │
│  └─────────────────────────────────────────────┘    │
│                                                     │
│  → 每步约 200-500 tokens (结构化后)                 │
│  → 保留 10-20 步 ≈ 5-10K tokens                    │
│                                                     │
├─────────────────────────────────────────────────────┤
│                                                     │
│  L3: EXTERNAL CAPSULES (冷数据，指针在，内容在外)     │
│  ─────────────────────────────────────────           │
│  完整内容不在窗口中。但是:                            │
│                                                     │
│  ⚠️ 关键设计: 不是"LLM 想起来了就去拉"               │
│                                                     │
│  框架强制推入规则 (不允许 LLM 跳过):                  │
│  ┌─────────────────────────────────────────────┐    │
│  │ 当 LLM 编辑 capsule 关联的文件时:            │    │
│  │   → 自动注入胶囊摘要到 L1 (框架推)           │    │
│  │   → LLM 不需要知道胶囊存在                   │    │
│  │                                             │    │
│  │ 当 LLM 执行与 capsule 冲突的操作时:          │    │
│  │   → 自动注入冲突警告 (框架推)                │    │
│  │   → "⚠️ 这个方案在 capsule:xxx 中已讨论并放弃 │    │
│  │      原因: 耦合太强"                         │    │
│  │                                             │    │
│  │ 当 capsule 包含 MANDATORY 标记时:            │    │
│  │   → 始终在 L1 保留摘要 (温度不衰减)          │    │
│  │   → 用于: 安全约束、法律约定、不可违背的规则  │    │
│  └─────────────────────────────────────────────┘    │
│                                                     │
│  指针格式 (在 L2 里):                                │
│  📦 capsule:fix_token_refresh                        │
│     ↑ 这些指针在 L2 中。LLM 可以看到。               │
│     但完整内容不在窗口。                              │
│     LLM 主动拉取: recall:context("fix_token_refresh") │
│     → 框架允许, 但框架不会等 LLM 想起来才给          │
│     → 框架在 LLM 编辑 auth.ts 时就推入摘要了         │
│                                                     │
└─────────────────────────────────────────────────────┘
```

## 4. 自主质询机制

这是第四个问题的答案：**"他有自我追问拿不准的能力吗？"**

**没有。所以我们给它。**

### 4.1 框架发起的质询

```
每步完成后，框架执行自主质询:

  ┌─────────────────────────────────────────────┐
  │ 1. 冲突检测                                  │
  │    当前 LLM 的方案是否与任何已知 capsule      │
  │    记录的决定冲突？                           │
  │    → 是 → 注入冲突警告到下一步的 L1          │
  │                                             │
  │ 2. 缺口检测                                  │
  │    LLM 编辑了文件 F，但 F 有 N 个关联         │
  │    capsule。LLM 是否引用了其中任何一个？       │
  │    → 没有 → 注入胶囊摘要到下一步的 L1        │
  │    → 框架不问 "你要不要看"，直接给。          │
  │                                             │
  │ 3. 一致性检测                                │
  │    LLM 当前编辑的文件是否修改了某个            │
  │    之前 capsule 中标记为 "已知限制" 的代码？   │
  │    → 是 → 注入 "⚠️ 你正在修改的代码与         │
  │       capsule:xxx 中的已知限制相关"           │
  │                                             │
  │ 4. 不完全信息标记                            │
  │    当前 LLM 的 action 涉及文件 F。F 的         │
  │    完整历史有 N 步在 L2/L3 中。               │
  │    框架标记: "注意: 关于此文件有 N 条          │
  │     历史上下文。其中关键摘要已推入 L1。"       │
  └─────────────────────────────────────────────┘
```

### 4.2 LLM 发起的质询（保留，不删除）

```
LLM 可以主动:
  recall:context("fix_token_refresh")  → 展开完整胶囊
  reflect()                            → 查看上下文健康度
  追问 "我是不是漏了什么?"              → 自然语言，不走工具

这些保留。但框架不会因为 LLM 没想起来就放过它。
```

要支持框架自动推入，胶囊需要标记：


## 6. 修正后的 L2 vs L3 区别

现在有实质性的区别了：

| | L2 (在窗口，压缩) | L3 (在窗口外，指针) |
|---|---|---|
| LLM 可见性 | 内容可见（压缩后） | **不可见**。只有指针 |
| 框架推入 | 已在窗口，无需推 | **按规则自动推摘要到 L1** |
| LLM 拉取 | `summarize:recent` 重新压缩 | `recall:context` 完整展开 |
| 触发拉取 | LLM 自己决定 | **框架强制：编辑关联文件→推摘要** |
| 冲突保护 | 可见所以一般不会冲突 | **框架主动检测冲突→自动告警** |
| MANDATORY 处理 | N/A (L2 内容终会降级) | 标记 MANDATORY 的胶囊永不降入 L3 |

## 7. 这样真的能更省更准吗？

### 7.1 省

**是的。但不是因为优雅，是因为框架替 LLM 做了判断。**

```
旧方案:  20K 文件内容 + 15K 日志 + 30K 历史全量 + 10K 噪音 = 75K
         → LLM 自己过滤 = 注意力分散 + 可能漏关键信息

新方案:  15K L0 + 35K L1(框架精选) + 8K L2(结构化历史) = 58K
         → 框架已过滤。LLM 不需要扫描日志和噪音。
         → 节省 17K/步 (23%)
         → 但 L3 的强制推入可能在关键步骤 +2-5K
         → 净节省约 15-20%
```

### 7.2 准

**比以前准。但不一定比"全量上下文"准。**

```
裸跑 (全量):                   新方案:
  LLM 看到所有信息                LLM 看到框架精选的信息
  → 可能被噪音分散               → 聚焦
  → 可能发现框架漏掉的联系        → 可能漏掉框架没推的关键信息 (L3 冷数据)
  
  准确率基准: 未知                准确率: 需要测试
  
  关键风险: 
  如果框架的推入规则有盲区 (如: 没有标记 capsule 关联某个文件),
  LLM 永远看不到那条信息 → 准确率下降。
```

### 7.3 诚实结论

**这个方案的核心假设是：框架的判断比 LLM 的注意力更可靠。**

这个假设需要被验证。验证方法：

```
对比测试:
  A 组: 全量上下文 (200K, 什么都不管)
  B 组: 框架精选 (L0+L1+L2, ≈60K)
  
  相同模型 + 相同任务 + 相同工具
  度量: pass rate + 总 token 消耗 + 单任务耗时

如果 B 组 pass rate 不比 A 组差 → 框架判断可靠 ✓
如果 B 组 pass rate 低于 A 组 → 框架推入规则有盲区 → 调整规则再测
如果 B 组 pass rate 大幅高于 A 组 → 噪音干扰确实是大问题 ✓✓
```

## 8. 与旧方案的对比

| 维度 | 应急驱逐 (70/85/90%) | 温度分层 (L1-L3) | **推挽双轨 (v2)** |
|------|----------------------|-------------------|---------------------|
| LLM 懒惰风险 | 低 (信息都在) | 高 (L3 盲区) | **中 (框架强制推)** |
| 准确率风险 | 低 | 高 (依赖 LLM 主动拉) | **中 (框架推, 但有盲区风险)** |
| 省 token | 中 (应急清, 锯齿状) | 高 (稳定) | **高 (稳定 + 精选)** |
| 框架复杂度 | 低 | 中 | **高 (推入规则 + 质询)** |
| 优雅程度 | 低 (panic 模式) | 高 | **中 (推挽是复杂的优雅)** |
| 是否需要 LLM 配合 | 不需要 | 不需要 (但期望 LLM 主动) | **不需要 (框架强制)** |

## 9. 实现优先级

```
P0: 推入规则引擎 (框架决定 LLM 需要什么)
    ├── 编辑文件 → 关联胶囊摘要自动推入
    ├── 错误/异常 → 必推
    └── 冲突检测 → 自动告警
    工期: 2 天

P1: 胶囊 auto_push_rules 生成
    ├── 打包时自动生成 edit_file / exec / propose_approach 触发规则
    └── 工期: 1 天

P2: 自主质询
    ├── 每步冲突/缺口/一致性检测
    └── 工期: 2 天

P3: 温度模型 (可选, 降优先级)
    ├── L1→L2→L3 的自然迁移
    └── 只有在推入规则稳定后才启用
    工期: 1 天

总计: 6 天 (P0+P1+P2)
```

## 10. 最后

> 一个好的上下文管理方案，应该是 LLM 感觉不到它在被管理。
>
> 它不需要自己判断 "我是不是该看了"。
> 它打开文件，该在的东西就在了。不该在的东西根本不在。
>
> 判断权在框架，不在 LLM。
>
> 这就是推挽双轨：推的是 LLM 可能需要但不一定想要的，拉的是 LLM 知道自己想要但不知道在哪的。
