# 四层冷热架构：设计推敲与优化

> 用户提案 vs 现有三层 → 逐层推敲 → 优化补充

---

## 1. 提案对比

| 层 | 现状 | 你提案 | 语义变化 |
|----|------|--------|---------|
| L1 | 活跃窗口（什么都往里塞） | 永久知识（习惯/方向/胶囊指针/查看工具） | "当前在用的"→"永远不该丢的" |
| L2 | 胶囊（压缩摘要） | 工作层（原本内容，当前对话） | "压缩后的"→"正在发生的" |
| L3 | 冷存原文（磁盘） | 压缩旧内容（很久之前的，但已压缩） | "原文"→"压缩后的旧对话" |
| L4 | — 不存在 — | 原文深存（原始旧对话，最底层） | 新增 |

**核心洞见**：你把 L1/L2 的语义反过来了，之前的设计确实混了"重要性"和"活跃度"两个维度。你的方案把它们分开——L1=%永久重要，L2=%当前活跃。

## 2. 推敲：每层边界在哪

### 2.1 L1（永久知识）——放什么不放什么

**该放的** ✅
- 用户偏好：语言、格式、回答风格
- 项目方向：架构决策、技术栈、核心约束
- 用户习惯：早上喜欢简短/晚上喜欢详细、工作流偏好
- L3 胶囊的**一行指针**（不是完整胶囊）：`[capsule:os_scheduler] 二叉堆优化, 延迟2ms→5ms`
- 召回工具的一般性描述（"如需历史细节，使用 recall 命令"）

**不该放的** ❌
- 完整胶囊正文（太长，放 L3）
- 昨天的具体调试记录（放 L3 压缩后）
- 任何可能过时的具体数据

**关键设计**：L1 必须有个硬预算——比如总窗口的 15%。因为用户会不断标"这个重要"，L1 会膨胀。膨胀后的处理不是删除，而是把最旧的 L1 条目自己压缩成胶囊，把胶囊指针留在 L1，正文移到 L3。

```typescript
// 伪代码
const L1_BUDGET = 0.15; // L1 最多占 15%
function compactL1() {
  while (L1.tokens > WINDOW * L1_BUDGET) {
    const oldest = L1.oldestUnpinned();
    const capsule = summarizeAndCapsule([oldest]);
    L3.store(capsule);           // 正文去 L3
    L1.replace(oldest, capsule.oneLiner()); // L1 只留一行指针
  }
}
```

### 2.2 L2（工作层）——跟 L1 的关系

L2 是 LLM 实际看到的东西。它由两部分构成：

```
L2 = L1内容（永久可见） + 当前对话窗口（活跃轮次）
```

**问题**：当前对话也在 L2 里，那 L2 满了怎么办？

答案：L2 满了不是"L2 爆了"，是**L2 里属于当前对话的那部分被逐步归档到 L3**。L1 的内容始终保留在 L2 中。

这引出一个重要规则：**LLM 看到的上下文永远是 L1 + L2中最近N轮的对话。L3/L4 在召回前不可见。**

### 2.3 L3（压缩旧内容）——何时触发迁移

你提的 20% 阈值想法是对的——等 70% 再管已经太晚了。但 20% 是什么的 20%？

| 解释 | 含义 | 好不好 |
|------|------|:---:|
| 总窗口的 20% | 30K 窗口 → 6K 就管 | ❌ 太激进，8-10 条消息就触发 |
| 非活跃内容的 20% | 距离当前话题 5+ 轮的内容占 20% | ✅ 合理 |
| 单次对话增量的 20% | 刚进来的大块内容（如工具输出 6K） | ✅ 合理 |

**建议**：不是固定 20%，是按两个维度组合触发——

```
管理触发器 = (token占比 > 20% AND 话题距离 > 3轮) OR (token占比 > 50%) OR (用户手动触发)
```

这样一条短消息不会触发，一个 6K 的工具输出会触发，一段 3 轮前的旧讨论会触发。

### 2.4 L4（原文深存）——L3 的下一站

L4 跟现在的 ContentStore 是一样的东西：磁盘上的原文。

**L3 → L4 迁移条件**：L3 胶囊超过一定数量/时间（比如 30 个胶囊或 7 天前的），把**胶囊对应的原文**迁移到 L4。L3 只保留压缩版。

**L4 存在的意义**：
- L3 是压缩后的小体积知识（~30 tokens/胶囊），LLM 召回便宜
- L4 是原始大体积数据（~400 tokens/条），只在需要精确原文时才查
- 分层降低召回成本：先搜 L3（快、便宜），不够再搜 L4（慢、贵、但精准）

## 3. 追问：你这个方案没有回答的问题

### 3.1 三层之间怎么流通？

```
         ┌──────────┐
  pin    │    L1    │ 永久知识，始终在 L2 里可见
  ↑      │ ≤15%窗口 │
         └────┬─────┘
              │ compactL1() → 旧 L1条目压缩 → L3，L1只留指针
              ↓
         ┌──────────┐
  当前   │    L2    │ 工作层 = L1指针 + 最近N轮对话
  对话   │          │
         └────┬─────┘
              │ 触发管理 → 旧对话压缩 → L3胶囊
              ↓
         ┌──────────┐
  召回   │    L3    │ 压缩旧内容（胶囊），语义召回
   ↑     │          │
         └────┬─────┘
              │ 胶囊原文 >7天 → 原文去L4，L3只留压缩
              ↓
         ┌──────────┐
  精确   │    L4    │ 原文深存（磁盘），FTS5搜索
  查找   │ ContentStore│
         └──────────┘
```

**关键**：
- L1→L3 是单向的（永久知识旧了也会归档，但留指针）
- L2→L3 是单向的（工作内容完成了就归档）
- L3→L2 是临时的（召回后注入到当前上下文，用完再回去）
- L3→L4 是单向的（原文太老了就去深存）
- L4→L2 是临时的（精确找回原文）

### 3.2 LLM 实际看到什么？

```
[SYSTEM] 你是一个 AI 助手。            ← L1: 系统提示
[L1] 用户偏好: 用中文回答，代码用TS    ← L1: 用户习惯
[L1] 项目: StructFocus 上下文管理      ← L1: 项目方向
[L1] 📦 capsule:os_scheduler 二叉堆优化 ← L1: 胶囊指针
[L1] 使用 recall("关键词") 查历史      ← L1: 工具提示
──────────────────────────────────────
[USER] 调度器延迟又涨了，帮我想想      ← L2: 当前对话
[AI] 上次是用二叉堆解决的...           ← L2: 当前对话
[USER] 但现在 200 进程了，log n 不够   ← L2: 当前对话
```

LLM 永远先看到 L1 的知识层，然后是 L2 的最近对话。L3 的完整胶囊和 L4 的原文不在 prompt 里——除非用户/Agent 主动调用 recall。

### 3.3 "不驱逐，只放冷"——但冷到什么时候算深存？

现在的问题是：驱逐 (`evict`) 这个词暗示删除。你的方案里没有删除，只有**降级**。但降级管道的终点是什么？

建议：**从不删除，只加深**。L4 可以用 SQLite FTS5 存，磁盘成本几乎为零。唯一的限制是用户手动清空 `.structfocus/` 目录。

## 4. 附加优化（现有管线没想到的）

### 4.1 L1 自动学习

L1 不应该只靠用户手动 pin。系统应该从对话中自动提取 L1 知识：

```
每次 flush 或 summarize 后:
  1. LLM 过一遍新的胶囊
  2. 问: "这些内容里有没有应该永久记住的用户偏好/项目决策？"
  3. 有 → 自动添加到 L1，标记 [auto-learned]
  4. 用户可以在 L1 面板里删除/修改 [auto-learned] 条目
```

这样 L1 会越来越准。用户不用显式说"记住我喜欢..."，系统自己从对话里学。

### 4.2 L1→L3 周期性重概括

L1 里的条目会过时。比如 "项目用 React 18" 在 3 个月后可能变成 "项目迁移到 React 19"。需要一个周期性的 L1 审计：

```
每 30 个胶囊或每 7 天:
  1. 把 L1 内容 + 最近胶囊喂给 LLM
  2. 问: "以下永久知识是否有过时/冲突？"
  3. 标记过时条目 → 压缩到 L3 → 从 L1 移除或更新
```

### 4.3 L2 分层显示

L2 是 LLM 看到的全部内容，但可以加一个**折叠机制**：

```
L2 内容按时间排序，但旧的 L1 条目和旧的对话轮次可以折叠:

[L1 永久知识 — 展开]
  • 用中文回答
  • 项目: StructFocus
  • 📦 capsule:os_scheduler (点击展开)

[L1 自动学习 — 折叠]
  • 用户偏好短回答 (auto, 3天前)
  • 不喜欢过度解释 (auto, 2天前)

[L2 当前对话]
  USER: 调度器延迟又涨了...
  AI: ...
```

折叠机制不是给 LLM 看的，是给**用户通过 UI 审查上下文**时用的。让用户一眼看到"LLM 看到什么"，信任感会大幅提升。

### 4.4 多 Agent 场景下的 L1 共享

如果 StructFocus 接到多 Agent 框架里（OpenClaw / CodeX），每个子 Agent 有自己的 L2/L3，但可以共享同一个 L1。

```
Main Agent L1 (共享)
├── 子 Agent A L2/L3/L4 (独立)
├── 子 Agent B L2/L3/L4 (独立)
└── 子 Agent C L2/L3/L4 (独立)
```

这等于把"项目知识"和"任务执行上下文"彻底分开。主 Agent 负责维护 L1 的正确性，子 Agent 只管 L2 里的任务。

### 4.5 管理触发条件——细化版

你说的"20% 开始管"加上我的补丁后：

```typescript
interface ManagementPolicy {
  // 用户可调，默认值
  softThreshold: 0.20;  // 非活跃内容 >20% → 开始标记、分组、预压缩评估
  hardThreshold: 0.50;  // 非活跃内容 >50% → 执行压缩归档 L2→L3
  emergencyThreshold: 0.85; // 总token >85% → 强制驱逐最冷条目到L4
  
  // 话题距离（辅助条件）
  topicDistance: 3;     // 距离当前话题 >3轮才算"非活跃"
  
  // 单次增量保护
  maxChunkBeforeManage: 4000; // 单条消息 >4K token 直接评估是否该压缩
  
  // 用户可随时调
  userOverride: "manual" | "auto";
}
```
