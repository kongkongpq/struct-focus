## 方法

- 数据集：100 条模拟「被驱逐上下文」条目，分 20 个主题簇（每簇 5 条），每条含独特关键词（中英文）+ 主题描述 + 跨主题噪声词。
- 查询：20 个（16 个精确关键词查询 + 4 个同义/模糊查询，用于暴露 BM25 无同义词感知的局限）。
- 金标准：每个查询标注其所属主题簇的 5 条为相关。
- 对比：ContentStore BM25（`search(query,{mode:'bm25',topK:5})`）vs 简单 `includes` 子串匹配（查询空白分词后全 token 子串命中）。
- 指标：Precision@5 = top5 中相关数 / 5；Recall@5 = top5 中相关数 / 相关总数(5)。

## 结论

- 在精确关键词查询上，BM25 与 includes 均能以满分的 P@5/R@5 召回相关条目；BM25 不劣于简单子串匹配，满足合格标准。
- 4 个同义/模糊查询的细分结果揭示了 BM25 的真实特性：
  - 「主从复制拓扑」「外部知识库问答」与正文**零词面重叠**，BM25 与 includes **双双失效**（均为 0.00），说明二者均无同义词/语义扩展能力。
  - 「协程调度框架」「缓存淘汰策略」因正文恰好含「调度」「淘汰策略」等子词，BM25 的 OR 式词面打分仍可部分命中（P@5/R@5=1.00），而 includes 因要求整短语全 token 匹配而失败（0.00）——BM25 比 includes-AND 更鲁棒。
- 改进方向（非必需）：引入同义词词典、子词/向量召回（hybrid 模式已预留 `mode:'hybrid'` 接口），可在零重叠的同义查询上进一步提升召回。
